# Контекст сессии

## Что делали
Разбирали причины, почему бот пару раз в день зависает (без краша, контейнер живой, но сообщения не пересылаются).

## Что уже сделано

### 1. Restart Policy в TrueNAS
Поменяли с `No` на `Unless Stopped` — теперь Docker будет автоматически поднимать контейнер если он упадёт.

### 2. Фикс asyncpg (`storage.py`)
Добавили таймауты на соединения с БД:
```python
command_timeout=30,
max_inactive_connection_lifetime=60,
```
Причина: при сетевом моргании TrueNAS соединение с Postgres умирало, asyncpg ждал ответа вечно → worker зависал.

### 3. Watchdog (`main.py`)
- Добавлен `import time` и глобальная переменная `last_activity = time.time()`
- Добавлена константа `WATCHDOG_TIMEOUT = 300` (5 минут)
- Добавлена функция `watchdog_loop()` — каждую минуту проверяет, был ли polling активен. Если нет — делает `sys.exit(1)`, Docker перезапускает контейнер.
- `last_activity` обновляется в `polling_loop` после каждого успешного цикла
- `watchdog_loop()` запускается как task в `main()`

### 4. Таймауты на Pyrogram API calls + перезапуск воркера (`main.py`)

**Проблема:** Бот исправно обнаруживал новые сообщения (polling работал, логи `📥 New message found`), но в канал ничего не отправлялось и логов `✅ Sent` не было.

**Причина:** `asyncio.create_task(worker_loop(...))` создавал задачу без отслеживания. Если воркер зависал на Pyrogram API call (`message.copy()`, `client.forward_messages()` и т.д.) или тихо падал с необработанным исключением — никто об этом не знал. `last_activity` обновлялся в `polling_loop`, поэтому watchdog не видел проблемы.

**Дополнительный симптом:** При попытке остановить контейнер через TrueNAS UI — контейнер не реагировал (event loop заморожен → SIGTERM не обрабатывался → потребовался `docker kill`).

**Исправления в `main.py`:**

1. **`API_TIMEOUT = 30`** — все Pyrogram вызовы обёрнуты в `asyncio.wait_for(..., timeout=30)`. При зависании бросается `TimeoutError`, воркер его ловит, логирует и идёт дальше.

2. **Отслеживание worker task** — `worker_task` сохраняется как глобальная переменная. Watchdog каждые 60 сек проверяет `worker_task.done()` и, если задача умерла, создаёт новую (без перезапуска всего процесса).

3. **`last_activity` перенесён в воркер** — обновляется только после реальной отправки сообщения. Polling по-прежнему обновляет его как fallback (чтобы watchdog не убивал бот в тихие периоды).

**Важно:** Сообщения, которые не были отправлены до рестарта, **не** помечались в Postgres как обработанные (`mark_processed` вызывается только после успешного send). После перезапуска бот корректно их подхватит.
